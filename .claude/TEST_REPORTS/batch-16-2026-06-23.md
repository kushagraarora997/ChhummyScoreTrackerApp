## What Was Fixed

### Root Cause: Double-Write Bug
When two devices confirmed the same round before either received the other's Firestore update:
- Each device called `confirmRound()` → generated a unique `nanoid()` as the round ID
- Each called `syncRound()` → wrote to Firestore under `round.id` (nanoid) as the document key
- Result: **two separate Firestore documents** for Round N (different IDs, same `number` field)
- `ingestCloudRound` only deduped by `r.id === round.id` → both docs got accepted → duplicate Round N in state

### Fix (4 files)

**`src/lib/firebaseSync.ts`**
- `syncRound`: changed Firestore doc key from `round.id` (nanoid) to `${round.sessionId}_${round.number}` (composite)
- `deleteRoundFromCloud`: signature changed from `(familyId, roundId)` to `(familyId, sessionId, roundNumber)`
- Both changes ensure at most ONE Firestore document exists per round number per session (last-write-wins)

**`src/db/operations.ts`**
- `deleteRound`: parameter changed from `id: string` to `round: Pick<Round, "id" | "sessionId" | "number">`
- Added `putRoundLocal(round)`: Dexie-only write (no Firestore re-sync), used when ingesting cloud rounds

**`src/store/useAppStore.ts`**
- `undoLastRound`: changed `deleteRound(last.id)` → `deleteRound(last)` (passes full round)
- `ingestCloudRound`: added number-based dedup guard as second check; switched to `putRoundLocal` to avoid unnecessary Firestore re-sync

## Bugs Found and Fixed

**Bug: Double-write race condition in multi-device games**
- Condition: Two devices confirm the same round before either receives the other's Firestore update
- Impact: Both devices end up with duplicate Round N entries in Zustand state
- Fix: Composite Firestore key (server-side prevention) + number-based dedup in `ingestCloudRound` (client-side safety net)

---

## Key Discoveries

1. **Composite key is self-sealing**: Even if two devices race to write Round N, Firestore's last-write-wins ensures only ONE document exists. New joiners via `pullFromCloud` never receive duplicates.
2. **Number-based dedup is the real-time safety net**: For the narrow window before the first device's onSnapshot reaches the second, the `ingestCloudRound` number check prevents accepting a duplicate.
3. **`putRoundLocal` avoids circular Firestore re-syncs**: Previously `ingestCloudRound` called `putRound` which re-synced to Firestore. Now it uses `putRoundLocal` (Dexie only) — cleaner and saves N×M Firestore write operations.
4. **DW4 proves defense-in-depth works**: Even with legacy/injected duplicate docs in Firestore (from pre-fix clients), the runtime dedup correctly ignores them.
